# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

**La Dolce Quiz — Classifica Evento** is a single-event quiz leaderboard. It has:
- A public TV display page (`/`) optimised for 1920×1080, polling Supabase every 5 seconds
- A password-protected admin backoffice (`/admin`) for deleting entries
- A public REST API (`POST /api/public/score`) that external quiz apps call to submit scores

## Commands

```bash
bun run dev       # start dev server (Vite + TanStack Start SSR)
bun run build     # production build (Nitro, targets Cloudflare Workers)
bun run lint      # ESLint
bun run format    # Prettier
```

There are no tests. The project uses **Bun** as the package manager — use `bun add` / `bun remove`, not npm/yarn.

## Architecture

### Framework: TanStack Start (SSR)

This is a **TanStack Start** app with server-side rendering, not a plain Vite SPA. Routes live in `src/routes/` and use file-based routing — `routeTree.gen.ts` is auto-generated by the router plugin on every dev/build run. Never edit it by hand.

| File pattern | URL |
|---|---|
| `index.tsx` | `/` |
| `admin.tsx` | `/admin` |
| `api/public/score.ts` | `/api/public/score` |
| `api/public/admin.login.ts` | `/api/public/admin/login` |
| `api/public/admin.delete.ts` | `/api/public/admin/delete` |

The root layout (`__root.tsx`) injects `QueryClientProvider` and the HTML shell. Adding pages requires creating a file here — do not create `src/pages/` (that's Next.js).

### Server routes vs UI routes

API endpoints use the `server.handlers` property on a TanStack route:

```ts
export const Route = createFileRoute("/api/public/score")({
  server: {
    handlers: {
      POST: async ({ request }) => { ... },
      OPTIONS: async () => new Response(null, { status: 204 }),
    },
  },
});
```

Files that must never reach the browser are named `*.server.ts` — Vite excludes them from the client bundle.

### Supabase

Two clients, used in different contexts:

- **`src/integrations/supabase/client.ts`** — anon key, safe for browser and SSR. Uses `import.meta.env.VITE_*` (client) with `process.env` fallback (SSR). Use for all reads (RLS permits public SELECT).
- **`src/integrations/supabase/client.server.ts`** — service role key, server-only. Bypasses RLS. Import only inside `*.server.ts` files or `server.handlers`. Used for INSERT and DELETE.

RLS policy: `leaderboard_entries` allows `SELECT` to `anon`/`authenticated`, but INSERT/DELETE are blocked at the DB level — only `service_role` (server routes) can write.

### Database schema

Single table: `public.leaderboard_entries`

| Column | Type | Notes |
|---|---|---|
| `id` | `uuid` | PK, `gen_random_uuid()` |
| `name` | `text` | NOT NULL |
| `score` | `integer` | NOT NULL, CHECK >= 0 |
| `created_at` | `timestamptz` | DEFAULT now() |

Index on `(score DESC, created_at ASC)` matches the leaderboard sort order (higher score wins; earlier timestamp breaks ties).

Migrations live in `supabase/migrations/`.

### Environment variables

| Variable | Where used | Notes |
|---|---|---|
| `VITE_SUPABASE_URL` | Client + SSR | Public |
| `VITE_SUPABASE_PUBLISHABLE_KEY` | Client + SSR | Public anon key |
| `SUPABASE_URL` | Server only | Same value, no VITE_ prefix |
| `SUPABASE_PUBLISHABLE_KEY` | Server only | Same anon key |
| `SUPABASE_SERVICE_ROLE_KEY` | Server only | Bypasses RLS — keep secret |
| `ADMIN_PASSWORD` | Server only | Used by `/api/public/admin/*` routes |

On Cloudflare Workers (production), `process.env` values are bound at request time — always read them **inside** a function or handler, never at module scope.

### Admin authentication

There is no session token or cookie. The admin password is:
1. Verified server-side via constant-time `safeEqual` (manual XOR loop, not `crypto.timingSafeEqual`, to stay edge-runtime compatible)
2. Stored in `sessionStorage` under `ladq_admin_pw` and re-sent with every mutating request

### Vite config

`vite.config.ts` uses `@lovable.dev/vite-tanstack-config`. This wrapper already includes TanStack Start, React, Tailwind, TypeScript paths, Nitro, and other plugins. **Do not add these plugins manually** — duplicates break the build.

### Design system

Custom CSS variables in `src/styles.css` define the Italian dark-teal aesthetic:

- `--cream` — text colour (`oklch(0.93 0.04 85)`)
- `--gold` — headings / accents (`oklch(0.82 0.12 80)`)
- `--teal-deep` — card backgrounds
- `--accent` — terracotta red, used for the #1 rank highlight
- `font-display` — Cormorant Garamond (serif) for titles and scores
- `font-sans` — Inter for body text

Use `var(--cream)`, `var(--gold)` etc. in Tailwind class literals (`text-[var(--cream)]`) rather than the semantic Tailwind tokens when the Italian palette is required.

The `.auto-scroll` CSS class animates vertical scroll using `--scroll-distance` CSS custom property. The leaderboard sets this dynamically via a `ResizeObserver` when content overflows the container.

### Score submission API

`POST /api/public/score` is open to the public with CORS (`Access-Control-Allow-Origin: *`). It accepts:
```json
{ "name": "string (1–30 chars)", "score": 0–1000000 }
```
Names pass through `isCleanName()` (`src/lib/profanity.ts`) — a word-boundary check against an Italian+English list — before insert.
